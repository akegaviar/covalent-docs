# Querying with Primer (Beginner)

In this section will we use following primer-query parameters and query operators:

|Name|Description|
|---|---|
|match|Filters the records to pass only the documents that match the specified condition(s).|
|sort|Sorts all input records and returns them in ascending or descending sorted order.|
|skip|Skips over the specified number of records|
|limit|Limits the number of records.|

## Primer syntax
In this section we will explain the syntax for:
- [Match](#match)
- [Sort](#sort)
- [skip](#skip)
- [Limit](#limit)



### Match

Match is used to filter records on a specific criteria provided in the json object. We can do match by two ways:
 - [Specific records](#specific-records)
 - [Embeded records](#embeded-records)

You can apply the equality condition to retrieve a specific record. The `<field>:<value>` expression in the query document achieves this:

```json
---
header: Query for a specific record
---
match=
{
    "field_1": "value_1"
}
```

#### Specific records

For example, in the previous case, we can filter out records to only return the transaction sent from `0x88fb3d509fc49b515bfeb04e23f53ba339563981`



```json
---
header: maching a specific record
---
match=
{
    "from_address":"0x6d903f6003cca6255d85cca4d3b5e5146dc33925"
}
```

URl: [https://api.covalenthq.com/v1/1/address/0xc0da01a04c3f3e0be433606045bb7017a7323e38/transactions_v2/?match={"from_address":"0x88fb3d509fc49b515bfeb04e23f53ba339563981"}](https://api.covalenthq.com/v1/1/address/0xc0da01a04c3f3e0be433606045bb7017a7323e38/transactions_v2/?&key=ckey_66c94c405aae4cb38d94092f634&match={%22from_address%22:%220x88fb3d509fc49b515bfeb04e23f53ba339563981%22})

#### Embeded records
Filtering can also be done on embedded records using the nested nature of json.

It's possible to `match` based on deeply nested embedded documents using the dot notation (`field.nested_field`). The syntax is extensible to infinite depth: `field.nested_field.second_nested_field`.
For example, following syntax will give all the log events whose name was equal to `Repay`



```json
---
header: matching an embedded record
---
match=
{
    "log_events.0.decoded.name": "Repay"
}
```

URL: [https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?match={"log_events.0.decoded.name":"Repay"}](https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?&key=ckey_66c94c405aae4cb38d94092f634&match={%22log_events.0.decoded.name%22:%22Repay%22})


### Sort
We can sort by simply selecting the element and determine the sort order, `1` for acsending and `-1` for decsending order.

**Note: Sorting is currently executed _after_ paginiation.**


```json
sort={
    "block_signed_at": 1
}
```

URL: [https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?sort={"block_signed_at":1}](https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?sort={%22block_signed_at%22:%221%22})

We can also sort by a nested value such as the amount being repayed in the `Repay` events.
We first need to select transactions that contain a `Repay` event using the match describe before and then sort by the repay values. We can use the match top level query parameters match and sort. Based on the precedence definition (section 1.0 note: #1), the match query will be executed first and then sort.

```json
---
header: using match and sort
---
match=
{
    "log_events.0.decoded.name": "Repay"
}
```

```json
&
sort=
{
    "log_events.0.decoded.params.3.value": -1
}
```
URL: [https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?match={"log_events.0.decoded.name":"Repay"}&sort={"log_events.0.decoded.params.3.value":-1}](https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?&key=ckey_66c94c405aae4cb38d94092f634&match={%22log_events.0.decoded.name%22:%22Repay%22}&sort={%22log_events.0.decoded.params.3.value%22:-1})

### Skip

we can skip the amount of records as a top level query.
for example, this will skip the first 5 records.
```json
skip=5
```

URL: [https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?skip=5](https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?&key=ckey_66c94c405aae4cb38d94092f634&skip=5)


### Limit

Limit can be used to put a check on how many records are needed to be returned.
Using this we can put limit of 6 after skipping first 5.

```json
---
header: akipping first 5 records
---
skip=5
```

```json
---
header: limiting the response to 6 records
---
&limit=6
```

URL: [https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?skip=5&limit=6](https://api.covalenthq.com/v1/1/address/0xA361718326c15715591c299427c62086F69923D9/transactions_v2/?&key=ckey_66c94c405aae4cb38d94092f634&skip=5&limit=6)

### About the data

The Compound governance contract emits events pertaining to a proposal's lifecycle and voting history of COMP token holders. Here are some of the events that are emitted through this contract:

* `ProposalCreated` - emitted when a new proposal is created
* `VoteCast` - emitted when a vote has been cast on a proposal
* `ProposalCanceled` - emitted when a proposal has been canceled
* `ProposalQueued` - emitted when a proposal has been queued in the Timelock
* `ProposalExecuted` - emitted when a proposal has been executed in the Timelock
